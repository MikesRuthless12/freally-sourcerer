# Sourcerer — Security Model

## Threat model

Sourcerer continuously indexes the user's filesystem and, with the content lens enabled, extracts text from documents. The index is one of the most personal artifacts on the user's machine — it represents a near-total map of their files. Threat model is rich.

| Threat | Mitigation |
|--------|-----------|
| Local attacker reads index on disk | Index files written with mode 0600 (Unix) / user-only ACL (Win); located under `%LOCALAPPDATA%` / `~/Library/Application Support/` / `~/.local/share/` (per-user, not system-wide). Optional at-rest encryption available behind a settings toggle (age-encrypted index, key wrapped by passphrase / Keychain / Credential Manager). |
| Malware exfiltrates the index | The daemon runs as the user, not SYSTEM/root. Malware running as the same user has the same access to the underlying files anyway — encrypting the index doesn't help. The architectural mitigation is keeping the index under the user's home, not in shared locations. |
| Malicious IPC client | Pipe / socket ACL'd to the user; server validates command schema strictly; no path arguments accepted that escape the configured index roots; no shell-out APIs. |
| HTTPS server abuse | OFF by default. When enabled: requires per-launch random token shown only in the UI; bound to localhost by default; CORS allowlist is opt-in. Listen on `127.0.0.1` / `[::1]` only unless the user explicitly binds an external interface. |
| URL protocol handler abuse (`sourcerer://`) | The handler opens the UI focused on a query — it does not execute paths or trigger downloads. Query strings are parsed by the same query parser as the search bar, with the same length / complexity bounds. |
| TOCTOU between journal event and index write | Event ordering preserved within a volume via the per-volume USN cursor / FSEvents stream-ID / inotify watch descriptor. Cross-volume events ordered by wall-clock with a clear "best-effort ordering" disclaimer. |
| **(Phase 1)** Stale USN cursor after journal recreation | The Windows subscriber persists a `(volume_serial, journal_id, next_usn)` triple under `%LOCALAPPDATA%\Sourcerer\cursors\`. On restart, if the volume's reported `UsnJournalID` no longer matches, the subscriber discards `next_usn` and bootstraps from the MFT; if `next_usn < FirstUsn` (journal wrap), the subscriber re-seeds to `FirstUsn`. Either case is logged at `info`. The cursor write is process-crash-safe (tmp-then-rename) but **not power-fail-safe** — a hard power loss between rename and the on-disk flush can lose the most recent cursor update; the subscriber recovers by replaying USN events from the previously-persisted position, or by re-bootstrapping the MFT if the journal was recreated. |
| **(Phase 1)** USN journal access requires elevation | Opening `\\.\<drive>:` for `FSCTL_QUERY_USN_JOURNAL` requires admin / SYSTEM. The packaged `Sourcerer-Indexd` Windows Service runs as LocalSystem by default; the foreground `sourcerer-indexd run --volume` mode surfaces a clear `Access is denied` error rather than silently failing. The service install path goes through SCM (`OpenSCManagerW` + `CreateServiceW`) and triggers the standard UAC prompt — no privilege-escalation primitives are introduced. |
| **(Phase 2)** Stale FSEvents stream cursor across volume changes | The macOS subscriber persists a `(root, device, last_event_id, fs_name, bootstrap_complete)` tuple under `~/Library/Application Support/Sourcerer/cursors/<root_hash>.json`. On open, if the watched root's current `stat.st_dev` no longer matches the persisted `device`, the cursor is discarded and the subscriber re-bootstraps; if the persisted `last_event_id` predates the volume's FSEvents history horizon, the kernel emits `MustScanSubDirs`, which triggers an inline subtree rescan. The cursor write is process-crash-safe (tmp-then-rename) but **not power-fail-safe** — same caveat as Phase 1's USN cursor; the `MustScanSubDirs` recovery path covers the lossy edge. |
| **(Phase 2)** macOS FSEvents agent runs without elevation | The launchd plist at `~/Library/LaunchAgents/io.mikeweaver.sourcerer.indexd.plist` is a per-user agent — `launchctl load -w` runs it as the logged-in user, never root. FSEvents per-user streams give us full coverage of the user's home tree without any kernel-mode hook. `Install` / `Uninstall` write only inside `~/Library/`; no system-wide files are touched. The plist's `ProgramArguments` is the absolute path of the running `sourcerer-indexd` binary captured at install time — there's no shell expansion, no PATH lookup, and the `service` subcommand argument is fixed. |
| Index tampering | Tantivy segment hashes verified on commit. Custom name index has an mmap'd checksum trailer verified on open; mismatch triggers WAL replay and surfaces a clear UI error rather than silently using corrupt data. |
| **(Phase 4)** Tampered or truncated `name.idx` / `name.suf` | Each file carries an 8-byte magic (`SRC-NAME`) + version trailer that `NameIndex::open` checks before mapping. On magic / version mismatch the open returns `IndexError::NameIndex(...)` rather than reading garbage; `Index::open` then re-derives the name index from the SQLite canonical store. Both files are written via tmp-rename so a crash mid-write never leaves a half-formed file in place — the previous good copy stays valid until rename succeeds. |
| **(Phase 4)** Hostile path content in `files.db` rows | SQLite columns are populated through `rusqlite::params!` parameter binding; no SQL is concatenated from path bytes. Path values cross the boundary as `to_string_lossy()` so non-UTF-8 bytes get the standard replacement-character treatment instead of producing arbitrary UTF-8 output downstream. The schema constrains `path` to `UNIQUE NOT NULL` so a duplicate-path injection cannot replace an existing row out-of-band. |
| **(Phase 4)** Index-root path traversal via crafted `JournalEvent` | `Index::apply` only writes into the `Index::open(root)`-rooted Tantivy directory, the `files.db` SQLite file, and `name.idx` / `name.suf` siblings. Event paths are never used as filesystem-write targets — they are stored as opaque keys + searchable values. Symlink / junction loops are out of scope here (handled in the journal subscribers in Phases 1–3). |
| **(Phase 4)** Unbounded indexer queue OOM under burst | `EventQueue` is bounded (default 10 000 events / Build-Guide spec). `try_push` returns `IndexError::QueueFull` so the journal subscriber back-pressures rather than the indexer growing without limit; `push_blocking` waits for room and honors close. The indexer cannot allocate beyond `capacity * sizeof(JournalEvent)` for the queue itself. |
| **(Phase 6)** Tampered or truncated `minhash.idx` | Same posture as Phase 4's `name.idx`. `minhash.idx` carries an 8-byte magic (`SRC-MNHS`) + version trailer and a header that records `K`, `BANDS`, `num_rows`, and `heap_len`; `SimilarityIndex::open` validates each on load and bounds-checks every row's heap slice before mapping. On magic / version / parameter / bounds mismatch the open returns `SimilarityError::Format(…)` rather than reading garbage; the caller (the indexd daemon) discards the file and rebuilds from upstream by walking `Index::store::iter_all` and re-upserting. The on-disk file is written atomically via tmp-rename so a crash mid-write never leaves a half-formed file in place — the previous good copy stays valid until rename succeeds. The bands map is *not* persisted; it is rebuilt on `open()` from the per-row signatures, so a tampered bands posting is structurally impossible. |
| **(Phase 6)** Hostile filename in `minhash.idx` heap | The heap stores the lower-cased filename bytes verbatim — the bigram MinHash never `eval`s, parses, or interprets the bytes beyond windowing them as `char` pairs. There is no SQL on the similarity path. The MinHash signature is purely arithmetic over a deterministic linear-hash family seeded by a hard-coded `MINHASH_SEED`; no user-controlled value reaches a hash-function constructor. A hostile filename can at worst inflate the bucket-postings vector for a band it shares with many corpus members; the per-query `candidate_cap` (default 1024) bounds the output unconditionally. |
| **(Phase 6)** Privacy of the similarity index | `minhash.idx` lives in the same per-user directory as `name.idx` and `files.db` — `%LOCALAPPDATA%\Sourcerer\index\` / `~/Library/Application Support/Sourcerer/index/` / `${XDG_DATA_HOME:-~/.local/share}/sourcerer/index/`. The signatures themselves are non-invertible (MinHash is a one-way summary) but the heap stores plain lower-cased filenames; treat it as personal data with the same posture as `files.db`. Optional at-rest encryption (Phase X) covers it identically. |
| Extractor crashes on hostile input (zip-bomb, malformed PDF, etc.) | Each extractor runs in a panicked-OK Tokio task with a per-document time budget (default 5 s) and memory ceiling (default 256 MB). Extraction failure marks the document `extracted: failed`, never crashes the daemon. |
| **(Phase 8)** Hostile PDF (encrypted / malformed / zip-bomb-shaped) | `PdfExtractor` reads at most `PDF_CAP_BYTES = 64 MiB` from disk before parsing; oversize files are truncated rather than mapped (no mmap on the PDF path). `pdf-extract` parse errors with "encrypted" / "password" in the message map to `ExtractError::Unsupported(...)` (the daemon routes them to a retry-with-prompt UI in Phase 12); other parse errors map to `ExtractError::Malformed(...)`. The extractor blocks on the whole-document parse — a heavy PDF that exceeds the sandbox time budget terminates by leaking the worker thread (the documented Phase-7 contract for non-cooperative extractors). Subprocess isolation for genuinely hostile PDFs is the Phase-13 evaluation. |
| **(Phase 8)** Hostile OOXML (docx / xlsx / pptx zip bomb / XXE) | All three OOXML extractors open the outer container as a `zip::ZipArchive`, which enforces individual-entry size + cumulative-uncompressed limits via the `zip` crate's defaults (rejects single-entry > 4 GiB + ratio bombs). docx + pptx parse the inner XML with `quick-xml = "0.39"` — quick-xml is a streaming parser with no DTD / external-entity resolution by design, so XXE is structurally impossible. xlsx goes through `calamine = "0.34"`, which parses sharedStrings + sheet streams with the same streaming posture. None of the three extractors call `unsafe`, dereference paths from XML attributes, or follow zip "../" entry names (the entry name is rendered into the search blob as a string only). |
| **(Phase 8)** Hostile archive (zip-slip, deep nested 7z, large tar) | `ArchivePeekExtractor` lists entries only — it never extracts a single byte to disk, so zip-slip (`../`/absolute-path entries) is structurally impossible. Per-archive cap `MAX_ENTRIES = 100_000` bounds the listing; a 1M-file tar surfaces the first 100k entries and stops. The 7z path uses `sevenz-rust2 = "0.21"`'s `Archive::read` (central-index walk), which never decodes pack streams. Cancel-flag checks fire every `CANCEL_CHECK_EVERY = 256` entries so the supervisor can interrupt a long enumeration. Entry names are written into the sink as `archive!path/inner size=N` strings — never used as filesystem paths. |
| **(Phase 8 — review pass)** Search-index poisoning via control characters in archive entry names / structured-data values | A hostile zip / 7z / tar entry could declare a name containing `\n`, `\r`, or `\0` and inject phantom rows into the search blob (one entry impersonating many); the same vector existed at the CSV / JSON / YAML scalar surface where quoted fields legitimately carry embedded newlines. `extractors::util::sanitize_inline` escapes those bytes (`\\n` / `\\r` / `\\0`) before format!, preserving search recall on the bytes themselves while keeping the line-per-record contract intact. Tabs and Unicode line separators (U+0085 / U+2028 / U+2029) pass through — the search index tokenizes on ASCII whitespace, not on every Unicode line break. Regressions: `entry_name_with_newline_is_sanitized`, `csv_field_with_newline_is_sanitized`, `json_string_value_with_newline_is_sanitized`. |
| **(Phase 8)** Hostile source code (tree-sitter parse-time DoS, malicious identifiers) | The code extractor reads at most `CODE_CAP_BYTES = 4 MiB` from disk before parsing; oversize files are truncated to a UTF-8 boundary (the trim loop pops bytes until `from_utf8` succeeds, never panics). `tree-sitter = "0.25"` is a pure-Rust runtime with no FFI to user code; grammars are statically linked and parse with the same memory posture as the runtime itself. Parse failures map to `ExtractError::Malformed`. Identifier extraction uses an allow-listed set of node kinds (`is_identifier_kind` / `is_string_kind` / `is_comment_kind`); arbitrary node names are not interpolated anywhere. The recursive tree walk fires a cancel-check every 1024 nodes; a deeply-nested adversarial parse tree gets cut by the sandbox time budget. |
| **(Phase 8)** Hostile structured-data (JSON bomb, billion laughs, deeply-nested CSV) | `StructuredDataExtractor` reads at most `STRUCTURED_DATA_CAP_BYTES = 16 MiB` from disk before parsing. JSON goes through `serde_json::from_slice` which has bounded recursion (the crate's documented stack-overflow guard); YAML through `serde_yaml_ng` (forked from serde_yaml — XML-billion-laughs is a YAML problem too, mitigated upstream by serde_yaml_ng's anchor/alias bounds); CSV through `csv = "1"` which is a streaming line reader with no global state. The `flatten_*` walks fire a cancel-check every 256 scalars / rows. None of the three formats are interpreted as code; the flattened key=value text never feeds back into a parser. |
| **(Phase 8)** Plain-text decoder DoS (lone surrogates / mixed encoding) | `PlainTextExtractor` caps reads at `PLAIN_TEXT_CAP_BYTES = 5 MiB` and only decodes once the magic bytes either match a recognized BOM or pass `looks_like_text` (rejects 2+ NULs in the head, requires valid UTF-8 after BOM strip). UTF-16 decode rejects odd-length input up-front; UTF-8 decode surfaces the first invalid sequence as `ExtractError::Malformed` (no `from_utf8_lossy` substitution, so a tampered file isn't silently re-encoded as `\u{FFFD}` strings). Truncation steps back to a UTF-8 boundary so the decoder never sees a half-codepoint. |
| **(Phase 8)** Tree-sitter dep version skew (`tree-sitter-dockerfile`) | `tree-sitter-dockerfile 0.2.0` is the only currently-published Dockerfile grammar and still binds to the tree-sitter 0.20 runtime. cargo's `links = "tree-sitter"` rule prevents mixing 0.20 and 0.25 in the same build graph, so we exclude the grammar from the dep set; Dockerfiles fall through to the plain-text extractor. The exclusion is documented inline in `crates/sourcerer-extractors/Cargo.toml` and tracked for revisit when the grammar rebases. No security impact — Dockerfiles are still searchable as plain text. |
| **(Phase 9)** Hostile audio file (malformed FLAC / MP3 / AAC / OGG / Vorbis / WAV / AIFF / M4A / ALAC) | `sourcerer-audio` decodes via `symphonia = "0.5"` — a memory-safe pure-Rust audio decoder with no FFI surface and no `unsafe` in our wrapper. Three cooperative defenses bound the worst case: (1) `MAX_CHANNELS = 64` caps the channel layout at the analyzer's front door so a malformed header that claims 1024 channels can't force enormous per-tap allocations inside ebur128; (2) `TimeBudgetSupervisor` runs every cache-driven extraction under `DEFAULT_AUDIO_TIME_BUDGET = 5 s` (matches the Phase-7 `Sandbox`'s default), flipping the cancel atomic via a detached supervisor thread that self-cleans on `Drop` once the extraction returns normally; (3) the cooperative cancel checks `Ordering::Acquire` per symphonia packet so a multi-hour decode aborts within tens of milliseconds of the flag flip. `SymphoniaError::ResetRequired` rebuilds the decoder *and* clears the cached `SampleBuffer` so the new decoder's `capacity()` doesn't trip a `copy_interleaved_typed` capacity-assertion panic on the next packet. `frames × ns / sample_rate` saturates via `try_into().unwrap_or(u64::MAX)` so a malformed header claiming `u64::MAX` frames cannot wrap the duration field. Subprocess isolation for genuinely hostile files is the same Phase-13 evaluation that covers the PDF non-cooperative-decode path. |
| **(Phase 9)** Audio cache JSON deserialization | `AudioCache::open` consumes `OnDisk { version, entries: HashMap<String, CacheEntry> }` via `serde_json::from_slice`. Schema-version mismatch drops the entries (`schema_version_mismatch_drops_entries` regresses the path) — a downgraded cache file from a future schema cannot inject malformed entries into a running daemon. The `lufs_serde` helper that round-trips `±∞ / NaN` rejects any sentinel string outside `{"-inf", "+inf", "inf", "nan"}` with a typed `serde_json::Error` — a hand-edited cache file that names a different sentinel surfaces `AudioError::Json` rather than producing a wrong value. Cache map keys are stored as JSON *values* and only ever round-trip back into a `HashMap` lookup — they are never used as filesystem-write targets, so a hostile cache file with `../../escape.flac`-shaped keys cannot escape the cache directory. The `flush()` path writes only to `self.persist_to` (constructor-supplied) plus the `<that>.json.tmp` sibling. |
| **(Phase 9)** Audio attribute query injection | `codec` flows from `codec_to_short_string` (a closed-enum mapping from symphonia's `CodecType` to fixed `&'static str` literals or `"unknown"`) into `AudioPredicate::Codec`'s `Vec<String>` and is compared with `eq_ignore_ascii_case`. No attacker-controlled bytes from the audio file content reach the predicate evaluator, and there is no SQL / shell / `eval` / regex sink for any audio attribute. All other AudioAttributes fields (`sample_rate`, `channels`, `bit_depth`, `duration_ns`, `lufs_*`, `peak_dbfs`, `silence_ratio`, `dynamic_range_lu`) are numeric. The query DSL's audio-modifier parsers (`parse_audio_lufs` / `_codec` / `_length` / `_rate` / `_silence` / `_dr`) validate user input strictly — out-of-range values, unknown units, and negative dynamic-range values surface `ParseError::InvalidModifierValue` rather than silently accepting. `cmp_op_f32`'s NaN guard returns `false` rather than trapping. |
| **(Phase 9)** Audio cache concurrency / TOCTOU | `AudioCache::get_with_opts` uses `Mutex<HashSet<PathBuf>> + Condvar` for single-flight extraction so two concurrent threads racing on the same cold path cannot both decode (the second waits on the condvar and re-runs `lookup(path, mtime_ns)` after wake-up). The in-flight slot is released via an RAII `InFlightGuard` so a panic inside `analyze_with_opts` still notifies waiters. mtime-keyed invalidation ties cached attrs to the requested mtime: a `lookup(path, mtime_ns)` whose stored entry's `mtime_ns` differs returns `None` and triggers re-extraction, so a journal `Modify` event observed after a query started cannot return stale attributes. The cache's atomic tmp-rename `flush()` uses the same trust-domain posture as Phase 4's `name.idx` and Phase 6's `minhash.idx`. |
| Symlink / junction loops during scan | Inode-set on Unix, junction-target tracking on Windows; loop detection skips offending entries with a logged warning. |
| Privilege-escalation via fanotify (Linux) | Sourcerer never runs as root. fanotify upgrade requires user-initiated polkit elevation; Sourcerer-Indexd remains a user unit. The polkit prompt explains exactly what's being elevated and why. |
| AGPL dependency drift (PDF rendering trap) | `cargo-deny` hard-blocks AGPL; CI fails on regressions. We ship pdf-rs (MIT), not mupdf or pdfium. |
| Update-channel attack | Updates signed with a fixed public key embedded in the binary; private key in CI secret; updates served from GitHub Releases over HTTPS only. |
| Dev-leaked telemetry | **Zero outbound network calls** by default. Auto-update is the single exception (1×/24 h to GitHub Releases); fully opt-out in settings. |
| Untrusted custom extractor (Wasm sidecar, Phase 12) | Custom extractors run in a wasmtime sandbox with no network, no filesystem-write, no clock beyond what's provided; a clear "this extractor is community-supplied" badge appears in the UI when one is enabled. |

## Out-of-scope (v0.19.84)

- Physical device theft. (Mitigation is full-disk encryption, not a feature of this app — though optional index encryption is available.)
- Hardware key-logging.
- Compromised OS kernel.
- Side-channel attacks against the on-disk index.

## Dependency policy

`deny.toml` allows: MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, ISC, CC0, Unlicense, Unicode-DFS-2016, Zlib, MPL-2.0.

`deny.toml` denies: GPL-2.0, GPL-3.0, AGPL-3.0, SSPL-1.0, BUSL-1.1, CC-BY-NC, CC-BY-SA.

`cargo-audit` runs weekly in CI. `cargo-vet` (Mozilla + Embark imports) gates releases.

## Code-signing posture

v0.19.84 ships unsigned MSI + NSIS (Windows), unsigned dmg + pkg (macOS), unsigned deb + rpm + AppImage (Linux). The install docs explain per-OS unsigned-installer prompts:

- **Windows.** SmartScreen reputation builds organically as downloads accumulate. Service install requires UAC (unavoidable, one-time at install).
- **macOS.** Gatekeeper requires right-click → Open on first launch. Documented prominently. Apple Developer Program ($99/yr) deferred to Phase X.
- **Linux.** No equivalent gating; AppImage runs directly; Flatpak / Snap submissions (Phase 13) bring their own signing.

Paid signing (Azure Trusted Signing for Win ~$10/mo, Apple Developer Program for macOS) is a documented Phase X upgrade once revenue justifies it. Hooks left commented in `release.yml`.

## Hardening flags

```toml
[profile.release]
panic = "abort"
strip = "symbols"
codegen-units = 1
lto = "fat"
overflow-checks = true
debug-assertions = false
```

Per-OS linker flags:

- **Windows.** Stack probes, CFG, /guard:cf, /DYNAMICBASE, /HIGHENTROPYVA. Daemon binary additionally: /CETCOMPAT.
- **macOS.** PIE, stack-check, hardened-runtime entitlements declared in Info.plist (no sandbox in v0.19.84 — full filesystem access is the entire product).
- **Linux.** PIE (`-pie`), stack-clash protection (`-fstack-clash-protection`), full RELRO (`-Wl,-z,relro,-z,now`), `_FORTIFY_SOURCE=2`.

## Privacy posture (user-facing)

The marketing claim is *zero outbound calls*. The single exception is auto-update, which is opt-out. The HTTPS server is opt-in. The custom-extractor framework's Wasm sandbox blocks network. No telemetry, no analytics, no AI services, no account, no sign-in. The first-run wizard explicitly tells the user this in plain language.
